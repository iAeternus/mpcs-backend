# 协同编辑领域设计与实现说明

本文档说明 `mpcs-backend` 与 `mpcs-frontend` 当前协同编辑领域的设计原理、接口协议、领域模型、OT 算法、WebSocket 通信方式，以及两个用户同时编辑同一个文件时如何做到实时更新。

适用代码范围：

- 后端：`mod-collaboration`
- 前端：`F:\Develop\java\mpcs\mpcs-frontend\src`

## 1. 目标与问题定义

协同编辑的核心目标是：

1. 多个用户可以同时进入同一文件的编辑会话。
2. 本地输入后，其他在线用户可以尽快看到更新。
3. 当两个用户基于同一版本同时编辑时，系统仍然能把操作合并成一致结果。
4. 文件保存后，后续进入协同页时可以用“文件内容 + 增量操作历史”恢复最新文档状态。

协同编辑的难点不在“广播字符串”，而在“并发修改的顺序一致性”：

- 用户 A 在位置 0 插入 `A`
- 用户 B 也在位置 0 插入 `B`

如果系统不做变换，最终不同客户端可能会得到不同文档：

- 某些客户端看到 `AB`
- 某些客户端看到 `BA`

这就是 OT（Operational Transformation，操作变换）需要解决的问题。

## 2. 总体架构

整个协同编辑链路由四部分组成：

1. 文件系统中的真实文件内容
2. 协同会话 `CollaborationSession`
3. 文本操作 `TextOperation`
4. WebSocket 实时消息通道

简化后的架构如下：

```text
文件预览/文件下载
        |
        v
  文件系统中的文件内容  <----- 保存时回写
        |
        v
CollaborationSession
  - version
  - baseVersion
  - operationHistory
  - activeUsers
  - cursors
        |
        v
WebSocket / REST
        |
        v
前端 useCollaborationEditor
  - 当前文本 content
  - pendingOps
  - inFlightOps
  - currentVersion
```

职责划分：

- 文件系统负责持久化最终文件内容。
- 协同领域负责维护“会话状态 + 未保存期间的增量操作历史”。
- OT 负责在并发情况下调整操作位置和长度。
- WebSocket 负责实时传输操作和确认消息。
- 前端负责本地 diff、乐观更新、远端操作重放以及与未确认本地操作的 rebase。

## 3. 领域模型

### 3.1 `CollaborationSession`

核心聚合根，代表一个文档对应的协同编辑会话。

关键字段：

- `id`：会话 ID
- `documentId`：文件 ID
- `documentTitle`：文件标题
- `parentFolderId`：文件父目录
- `version`：当前会话版本与文档长度
- `baseVersion`：文件最近一次保存时对应的协同版本
- `activeUsers`：当前在线协同用户
- `cursors`：用户光标位置
- `operationHistory`：自会话创建以来的操作历史
- `expiresAt`：会话过期时间

关键语义：

- `version` 每接收一个有效操作就加 1。
- `operationHistory[n]` 对应版本区间中的一条操作。
- `baseVersion` 表示“文件系统内容已经包含了多少条协同操作”。

因此：

- 当前文件展示内容 = 文件系统内容 + `operationHistory[baseVersion..latest]`

### 3.2 `SessionVersion`

用于维护：

- `version`
- `documentLength`

当插入发生时，`documentLength += insert.length`。  
当删除发生时，`documentLength -= delete.length`。

### 3.3 `TextOperation`

协同编辑的最小操作单元。

字段：

- `id`
- `type`：`INSERT | DELETE | RETAIN`
- `position`
- `content`
- `length`
- `userId`
- `clientVersion`
- `timestamp`

约束：

- `INSERT` 使用 `position + content`
- `DELETE` 使用 `position + length`
- `RETAIN` 当前主要用于扩展，现有主流程中不是核心操作

`clientVersion` 非常关键。它表示：

- 客户端发出这条操作时，自己认为当前文档是基于哪个服务端版本

**服务端必须用这个版本来判断哪些操作是“客户端没见过的并发操作”。**

### 3.4 `CollabUser` 与 `CursorPosition`

分别表示：

- 会话参与者
- 参与者的光标/选区状态

它们解决的是“谁在线、光标在哪”，不直接参与 OT。

## 4. 版本模型与 `baseVersion`

协同会话中同时存在两个版本概念：

### 4.1 `version`

会话当前总版本。

例如：

- 初始 `version = 0`
- 第 1 个操作后 `version = 1`
- 第 2 个操作后 `version = 2`

### 4.2 `baseVersion`

文件最近一次保存到文件系统时的版本。

例如：

1. 文件内容最初为 `HAMLET...`
2. 用户发生了 10 次协同编辑操作，此时 `version = 10`
3. 点击保存，把当前内容写回文件系统
4. 同时把 `baseVersion` 更新为 `10`

此后重新打开协同页时：

- 先从文件系统读出最新文件内容
- 再只加载 `fromVersion = baseVersion` 之后的新操作

这样可以避免把已经保存进文件的历史操作再回放一遍。

```json
{
  "type": "operation",
  "sessionId": "CLSxxx",
  "serverVersion": 13,
  "operation": {
    "type": "INSERT",
    "position": 6,
    "content": "X",
    "length": 1,
    "userId": "USR_A",
    "clientVersion": 12
  }
}
```

这里广播的是：

- **经过 OT 变换后的最终操作**
- 不是客户端原始提交的操作

这是保证所有客户端与服务端收敛一致的关键点。

## 7. OT 算法原理

OT 的核心思想是：

- 如果客户端操作 `C` 是基于旧版本生成的
- 服务端已经接受了一些客户端没见过的并发操作 `S1, S2, S3`
- 那么服务端不能直接应用 `C`
- 而是要先把 `C` 变成 `C' = transform(C, S1, S2, S3)`

然后再把 `C'` 应用到当前文档上。

## 7.1 为什么必须只 transform “并发操作”

如果客户端的 `clientVersion = 10`，服务端当前 `version = 12`，说明：

- 客户端已经知道并基于版本 10 的文档编辑
- 服务端多出来的只是第 11、12 两条操作

因此只应该取：

- `operationHistory[10..latest]`

而不是：

- `getRecentOperations(100)`

如果把客户端已经见过的旧操作再 transform 一次，会导致位置重复偏移，内容错位。

## 7.2 当前实现的 4 类基本变换

实现位置：

- `OperationTransformer`

当前处理四种组合：

1. `INSERT` vs `INSERT`
2. `INSERT` vs `DELETE`
3. `DELETE` vs `INSERT`
4. `DELETE` vs `DELETE`

### 7.2.1 Insert vs Insert

场景：

- 两个用户都在插入内容

原则：

- 如果客户端插入位置在服务端插入之前，不变
- 如果在之后，位置向后平移 `serverInsert.length`
- 如果位置相同，用 `userId` 作为稳定 tie-breaker

这样可以保证两个客户端最终顺序一致。

#### 7.2.1.1 为什么判断条件必须是 `clientPos > serverPos`

这里比较的是“插入点的先后关系”，不是“客户端插入点是否落入服务端新插入文本形成的区间”。

服务端操作为：

- `S = insert(serverPos, serverText)`
- 其中 `serverLen = length(serverText)`

客户端操作为：

- `C = insert(clientPos, clientText)`

对 `C` 做 OT 时，只需要判断 `clientPos` 相对于 `serverPos` 的位置：

1. `clientPos < serverPos`
   说明客户端插入点在服务端插入点之前，服务端插入不会影响这个锚点，因此位置不变。
2. `clientPos > serverPos`
   说明客户端插入点在服务端插入点之后。服务端先插入长度为 `serverLen` 的内容后，原来位于 `clientPos` 的插入点必须整体后移 `serverLen`。
3. `clientPos == serverPos`
   说明两个插入发生在同一个锚点上，此时不能靠区间判断，必须使用稳定 tie-breaker（当前实现为 `userId`）来决定先后顺序。

因此正确条件是：

```text
if clientPos < serverPos: keep
if clientPos > serverPos: shift by serverLen
if clientPos == serverPos: tie-break
```

而不是：

```text
if clientPos > serverPos + serverLen
```

后者把插入错误地当成了“覆盖 `[serverPos, serverPos + serverLen)` 的一个区间”。但插入操作并不会覆盖原文中的某一段字符，它只是在 `serverPos` 这个锚点前插入新文本，所以 OT 里并不存在“客户端插入点落在服务端插入区间内部”这一类语义。

#### 7.2.1.2 错误条件为什么会导致位置被误判

考虑下面这个顺序编辑场景：

- 原文：`abcde`
- 服务端先执行：`insert(0, "Hello")`
- 客户端随后基于新文档 `Helloabcde` 在位置 `5` 执行：`insert(5, "!")`

这里 `clientPos = 5` 的含义是：“在 `Hello` 后面插入 `!`”。  
因为这个插入点显然在服务端插入点 `0` 之后，所以应当满足 `clientPos > serverPos`，从而把位置平移到 `10`，最终结果为：

```text
Helloabcde!
```

如果使用错误条件 `clientPos > serverPos + serverLen`，则会得到：

- `5 > 0 + 5` 为 `false`

这会把一个“正常的顺序插入”误判成“同点插入/重叠情况”，导致它不被平移，或者被错误地交给 tie-breaker 处理。最终表现就是位置计算错误，出现我们这次回归里看到的 `5` 被错误保留下来、后续接口测试断言失败的问题。

#### 7.2.1.3 数学正确性证明

设原始文档为 `D`，服务端插入操作为：

```text
S = insert(p, s)
```

其中 `|s| = L`。

客户端插入操作为：

```text
C = insert(q, c)
```

OT 的目标是构造变换后的客户端操作 `C' = transform(C, S)`，使其满足收敛性：

```text
apply(apply(D, S), C') = apply(apply(D, C), S')
```

其中 `S'` 是对称变换后的服务端操作。

对于 `C'` 的位置 `q'`，分类讨论如下：

1. 当 `q < p`

   客户端插入点位于服务端插入点之前。先执行 `S` 不会改变该锚点之前的相对位置，因此：

   ```text
   q' = q
   ```

2. 当 `q > p`

   服务端已经在位置 `p` 前插入了长度为 `L` 的文本，因此原来位于 `q` 的插入点在新文档中的位置必须整体后移 `L`：

   ```text
   q' = q + L
   ```

3. 当 `q = p`

   两个插入共享同一个锚点，无法单靠位置比较决定先后，只能由稳定规则决定。例如当前实现按 `userId` 排序，谁更小谁保留原位，另一方后移 `L`。

注意，上述证明中从头到尾只需要比较 `q` 和 `p`，并不需要比较 `q` 和 `p + L`。原因是：

- `p` 是插入锚点
- `L` 只影响“锚点之后的位置整体平移多少”
- 它不定义一个可供插入操作“落入内部”的原文区间

因此，`INSERT vs INSERT` 的正确判定一定是：

```text
q < p, q > p, q = p
```

而不是：

```text
q < p, q > p + L, otherwise overlap
```

后者不符合插入操作的数学语义，也会破坏 OT 收敛所需的正确坐标变换。

### 7.2.2 Insert vs Delete

场景：

- 客户端要插入的地方，服务端可能已经删掉了一段内容

原则：

- 如果插入点在删除区间前，不变
- 如果在删除区间后，位置前移 `delete.length`
- 如果落在删除区间内，插入点移动到删除起点

### 7.2.3 Delete vs Insert

场景：

- 客户端要删除一段内容，但服务端已经在这段区域前后或中间插入了新内容

原则：

- 如果插入发生在删除前，删除起点后移
- 如果插入发生在删除中间，删除长度扩大

这样客户端“想删的语义范围”仍然成立。

### 7.2.4 Delete vs Delete

场景：

- 两边都在删除

原则：

- 若区间不重叠，则简单前移/保持
- 若区间重叠，则根据重叠量缩短或消除删除

如果服务端删除已经完全覆盖客户端删除，则客户端删除退化为 no-op。

## 7.3 批量变换

服务端批量处理时，会对每条客户端操作按顺序执行：

```text
for clientOp in batch:
    concurrentOps = operationsSince(clientOp.clientVersion)
    transformed = transform(clientOp, concurrentOps)
    append(transformed)
```

注意：

- 不能先拿一份固定的 `serverOps` 给整个 batch 复用
- 因为 batch 中前一条操作落库后，会改变后续操作所面对的服务端版本

## 8. 后端处理流程

以 WebSocket 批量提交为例：

1. 前端发送 `operation_batch`
2. 服务端取出会话 `CollaborationSession`
3. 校验用户是否在会话内、会话是否过期
4. 对 batch 中每一条操作：
   - 按 `clientVersion` 获取 `getOperationsSince(clientVersion)`
   - 调用 `OperationTransformer.transform`
   - 得到 `transformedOp`
   - 把 `transformedOp` 写入 `operationHistory`
   - 更新 `version`
   - 向其他在线用户广播 `transformedOp`
5. 向发送者返回 `operation_ack`

这样服务端始终是权威状态源。

## 9. 前端如何实现协同编辑

前端核心 Hook：

- `src/hooks/useCollaborationEditor.ts`

前端 OT 工具：

- `src/utils/collaboration-ot.ts`

## 9.1 初始化流程

进入协同编辑页时：

1. 获取当前用户信息
2. 按文件 ID 查找或创建会话
3. 调用 `joinSession`
4. 从文件系统读取文件内容
5. 读取 `baseVersion`
6. 调用 `getOperationHistory(sessionId, baseVersion)`
7. 按顺序把这些增量操作重放到文件内容上
8. 建立 WebSocket

这样进入协同页时看到的内容就是：

- 文件真实内容 + 未保存的协同增量

## 9.2 本地编辑如何生成操作

前端不会把整个文本每次都上传，而是会比较：

- `previousContent`
- `newContent`

然后生成最小差量操作。

当前实现使用“公共前缀 + 公共后缀”方式，把一次编辑折叠为：

- 0 或 1 个 `DELETE`
- 0 或 1 个 `INSERT`

例如：

```text
old: HelloWorld
new: Hello X World
```

可能被折叠为：

- `INSERT(position=5, content=" X ")`

这比旧的逐字符 diff 更稳定，也能显著减少日志和消息量。

## 9.3 为什么前端要维护 `pendingOps` 与 `inFlightOps`

前端本地会有两类尚未被服务端确认的操作：

- `pendingOps`：还没发送出去
- `inFlightOps`：已经发出，正在等待 ack

这两个队列的作用是：

- 本地输入可以立即更新界面
- 远端操作到达时，前端可以把远端操作与本地未确认操作做 rebase

否则会出现：

- 远端操作直接套用到旧位置
- 本地未确认操作再应用一次
- 最终内容错位

## 9.4 收到远端操作时前端做什么

前端收到服务端广播的 `operation` 后：

1. 如果是自己发出的操作，忽略
2. 取出 `inFlightOps + pendingOps`
3. 对远端操作和本地未确认操作逐个做 rebase
4. 把 rebased 后的远端操作应用到当前文本
5. 用 rebased 后的本地操作替换本地未确认队列
6. 更新 `currentVersion`

这一步保证：

- 远端更新能实时显示
- 本地还没确认的编辑不会被覆盖

## 10. 两个用户同时编辑一个文件时如何做到实时更新

下面用完整时序说明。

假设初始文档是：

```text
Hello
```

服务端版本：

```text
version = 0
```

用户 A 和用户 B 同时打开文件。

### 10.1 双方初始化

1. A 读取文件内容 `Hello`
2. B 读取文件内容 `Hello`
3. A 和 B 都拿到 `currentVersion = 0`
4. 两人都建立 WebSocket

### 10.2 A 与 B 同时编辑

假设：

- A 在位置 0 插入 `A`
- B 在位置 0 插入 `B`

两边本地都会立即变更：

- A 本地看到 `AHello`
- B 本地看到 `BHello`

同时发送给服务端：

- A: `INSERT(0, "A", clientVersion=0)`
- B: `INSERT(0, "B", clientVersion=0)`

### 10.3 服务端先处理 A

服务端当前没有比 `clientVersion=0` 更新的历史，因此：

- `A' = A`

写入后：

- 文档逻辑版本 `version = 1`
- 广播 `A'`
- 给 A 返回 `ack(serverVersion=1)`

### 10.4 服务端再处理 B

此时服务端版本已是 1，且 `operationHistory` 中多了一条 A 的插入。

服务端计算：

- `serverOps = getOperationsSince(0)`，只取到 A 的那条操作
- `B' = transform(B, A)`

因为两人都在位置 0 插入，系统按 `userId` tie-breaker 决定最终顺序。

若 A 的 `userId < B 的 userId`，则：

- A 保持在 0
- B 平移到 1

最终 B 被写成：

- `INSERT(1, "B")`

写入后：

- `version = 2`
- 服务端广播 `B'`
- 给 B 返回 `ack(serverVersion=2)`

### 10.5 A 收到来自 B 的广播

A 先前本地内容是：

```text
AHello
```

现在收到远端操作：

```text
INSERT(1, "B")
```

A 直接应用后得到：

```text
ABHello
```

### 10.6 B 收到来自 A 的广播和 ack

B 本地之前看到的是：

```text
BHello
```

但是 B 还有一条未确认本地操作，所以前端不能简单直接套广播。  
它会把：

- 远端 A 的操作
- 本地未确认的 B 操作

做 rebase，然后再应用。

最终 B 也会得到：

```text
ABHello
```

这样 A、B、服务端都收敛到同一结果。

## 11. 为什么系统能做到“实时”

实时更新来自以下机制的组合：

1. 本地输入先乐观更新 UI
2. 操作通过 WebSocket 立刻发送
3. 服务端立刻变换并落库
4. 服务端立刻广播给其他在线用户
5. 其他用户前端立即重放远端操作

因此用户体感上是：

- 自己输入立即可见
- 其他人输入在网络延迟允许范围内几乎实时可见

## 12. 当前实现的关键正确性约束

以下规则必须保持：

1. 服务端 transform 只能基于 `clientVersion` 之后的并发操作。
2. 服务端广播必须广播 `transformedOp`，不能广播原始操作。
3. 发送者收到的是 `operation_ack(serverVersion=...)`。
4. 前端必须处理远端 `operation` 消息，不能只处理 ack。
5. 前端本地 diff 不能使用逐字符洪泛算法。
6. 前端必须维护未确认本地操作，并在远端消息到达时 rebase。
7. 文件保存成功后必须更新 `baseVersion`。

其中任意一条出错，都会导致：

- 文本错位
- 重复字符
- 实时不同步
- 历史回放污染

## 13. 测试策略

当前已经补充和修正的测试重点包括：

### 13.1 OT 变换单测

- 同位置插入是否稳定收敛
- 插入与删除交错时位置是否正确
- 删除区间被覆盖时是否退化为 no-op
- batch 处理中零长度删除是否被过滤

### 13.2 接口级测试

- 只对并发操作做 transform
- 同 `baseVersion` 的并发插入是否能重放出一致结果
- `history(fromVersion)` 是否返回正确区间
- `baseVersion` 更新后，后续只回放新增操作

## 14. 典型问题与排查建议

### 14.1 进入协同页就乱码

优先检查：

1. `baseVersion` 是否正确更新
2. `history(fromVersion)` 是否取对了范围
3. 历史操作是否已经被错误 OT 污染

### 14.2 后端日志大量打印 `submitted operation`

优先检查：

1. 前端是否把一次编辑拆成过多逐字符操作
2. 前端是否反复把相同内容重新入队
3. ack 后是否正确清空 `inFlightOps`

### 14.3 两个客户端内容不同步

优先检查：

1. 服务端广播的是不是 `transformedOp`
2. 前端是否真正处理了远端 `operation`
3. 前端收到远端消息后有没有对本地未确认操作做 rebase

## 15. 后续可演进方向

当前实现已经可以支持基础实时协同，但仍有可升级空间：

1. 用更标准的文本 OT 模型替代当前简化版 `insert/delete` 变换。
2. 在前端引入更强的编辑器模型，而不是 `TextArea`。
3. 为光标同步增加更完整的广播与渲染。
4. 增加 WebSocket 集成测试，覆盖真实广播链路。
5. 对 `operationHistory` 做定期压缩或快照，避免长会话无限增长。
6. 在保存时生成服务器端文档快照，降低历史回放成本。

## 16. 一句话总结

协同编辑的本质不是“把别人输入的字符发给我”，而是：

- 每个客户端都把自己的编辑表示为操作
- 服务端用 OT 把并发操作变换到同一时空坐标系
- 前端再把远端操作与本地未确认操作做 rebase
- 最终所有参与者收敛到同一份文档

这就是当前协同编辑领域能够支持多用户实时编辑同一文件的核心原理。
